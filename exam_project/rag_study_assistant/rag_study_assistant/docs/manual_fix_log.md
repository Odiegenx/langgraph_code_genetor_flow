# Manual fix log

This file documents what had to be changed after the LangGraph workflow generated the Local RAG Study Assistant.

## Source of the generated project

- Generated run: `results/runs/student_assist_with_rags_06`
- Exam copy: `exam project/rag_study_assistant/rag_study_assistant`
- Input request: `input_requests/rag_study_assistant/request.md`
- Generated workflow inputs: `generated_inputs/local-rag-study-assistant`
- Generated review evidence: `workflow_review.md`
- Generated validation evidence: `site_validation_output.txt`

## What the workflow generated correctly

- A Flask application structure with a browser UI.
- A local RAG structure with ingestion, retrieval, prompt building, and Ollama client modules.
- A 4T prompt template in `prompts/rag_4t_prompt.md`.
- A validation script and documentation artifacts.
- A local-first design that uses Ollama instead of external API keys.

## What had to be fixed manually

The generated review file mentioned a likely import mismatch: `app.py` imported `ask_ollama`, while the Ollama module exposed `send_prompt_to_ollama`. The same kind of mismatch existed between `app.py` and the generated RAG modules:

- `app.py` expected `ingest_documents`, but `rag/ingest.py` only exposed `process_documents`.
- `app.py` expected `retrieve_chunks`, but `rag/retrieve.py` exposed `retrieve_context`.
- `app.py` expected `build_rag_prompt`, but `rag/prompt_builder.py` exposed a `PromptBuilder` class.
- `app.py` expected `ask_ollama`, but `rag/ollama_client.py` exposed `send_prompt_to_ollama`.

These issues were fixed by adding compatibility functions to the generated modules instead of rewriting the whole application.

## Validation fixes

The validation script was too strict about the 4T prompt format. It required literal labels like `Traits:`, but the generated prompt used Markdown headings like `## Traits`. The validator was changed to accept Markdown headings with or without a colon.

The validation script also wrote emoji output without explicitly using UTF-8. On Windows this can fail under the default `cp1252` encoding. The validation output file is now opened with `encoding="utf-8"`.

The validation script now returns an explicit exit code:

- `0` when all validations pass.
- `1` when one or more validations fail.

## Dependency setup

The generated project required Flask, pypdf, and requests. These were installed into the existing project virtual environment using:

```powershell
..\..\..\.venv\Scripts\python.exe -m pip install -r requirements.txt
```

This connected directly to the generated `requirements.txt` file.

## Connection to generated workflow information

These fixes were not random manual rewrites. They were based on artifacts produced by the workflow:

- `workflow_review.md` pointed to a possible Ollama function import mismatch.
- `site_validation_output.txt` showed that validation needed to be rerun in the copied project.
- The generated task and contract required local RAG, 4T prompt engineering, source-grounded answers, and a validation script.
- The generated project already contained the right modules, but the public function names did not line up with `app.py`.

The manual work therefore focused on making the generated components work together while preserving the architecture and requirements generated by the workflow.


If dependencies are missing, install them first:

```powershell

```

Then run the app:

```powershell
..\..\..\.venv\Scripts\python.exe app.py
```

Ollama must be running on `http://localhost:11434`, and the configured local model must be available.
